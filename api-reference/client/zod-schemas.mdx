The `@workspace/api-zod` package provides Zod schemas and TypeScript interfaces generated directly from the OpenAPI spec. The API server uses these schemas for runtime response validation before sending data to clients.

## Installation

`@workspace/api-zod` is a private monorepo package. Add it to another package's `package.json`:

```json
{
  "dependencies": {
    "@workspace/api-zod": "workspace:*"
  }
}
```

## Exports

The package root (`"."`) re-exports everything from both the generated schemas and the generated types:

```ts
// lib/api-zod/src/index.ts
export * from "./generated/api";   // Zod schemas
export * from "./generated/types"; // TypeScript interfaces
```

## Server-side usage

The API server imports `HealthCheckResponse` directly and calls `.parse()` to validate the response payload before returning it. This ensures the response always conforms to the OpenAPI contract at runtime.

```ts
// artifacts/api-server/src/routes/health.ts
import { Router, type IRouter } from "express";
import { HealthCheckResponse } from "@workspace/api-zod";

const router: IRouter = Router();

router.get("/healthz", (_req, res) => {
  const data = HealthCheckResponse.parse({ status: "ok" });
  res.json(data);
});

export default router;
```

<Note>
  If the payload passed to `.parse()` does not match the schema (e.g. `status` is missing), Zod will throw a `ZodError` before the response is sent. This acts as a compile-time and runtime contract between the server implementation and the OpenAPI spec.
</Note>
