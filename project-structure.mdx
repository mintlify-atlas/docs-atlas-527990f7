EduLearn Hub is a **pnpm workspaces monorepo**. All packages live under a
single repository root, share a common lockfile, and can reference each other
via `workspace:*` version specifiers without publishing to a registry.

## Directory tree

```text
edulearn-hub/
├── artifacts/              # Deployable applications
│   └── api-server/         # Express API server
├── lib/                    # Shared libraries
│   ├── api-spec/           # OpenAPI spec + Orval codegen config
│   ├── api-client-react/   # Generated React Query hooks
│   ├── api-zod/            # Generated Zod schemas from OpenAPI
│   └── db/                 # Drizzle ORM schema + DB connection
├── scripts/                # Utility scripts (single workspace package)
│   └── src/                # Individual .ts scripts
├── pnpm-workspace.yaml     # Workspace glob patterns
├── tsconfig.base.json      # Shared TS options (composite, bundler resolution, es2022)
├── tsconfig.json           # Root TS project references
└── package.json            # Root package with hoisted devDeps
```

## Top-level directories

| Directory | Purpose |
|---|---|
| `artifacts/` | Deployable applications — anything that runs as a server or standalone process |
| `lib/` | Shared libraries consumed by artifacts and each other; never deployed on their own |
| `scripts/` | One-off utility scripts; each `.ts` file in `src/` gets its own npm script entry |

## Workspace packages

<CardGroup cols={2}>
  <Card title="api-server" icon="server">
    **`artifacts/api-server`** · `@workspace/api-server`

    Express 5 API server. Routes live in `src/routes/` and use `@workspace/api-zod`
    for validation and `@workspace/db` for persistence. The entry point reads
    `PORT` and starts the listener; the app module mounts CORS, body parsing, and
    all routes under `/api`.

    **Scripts:** `dev`, `build`, `typecheck`
  </Card>

  <Card title="db" icon="database">
    **`lib/db`** · `@workspace/db`

    Database layer built on Drizzle ORM + PostgreSQL (`pg`). Exports a Drizzle
    client instance and all schema models. `drizzle.config.ts` requires
    `DATABASE_URL`. Schema is pushed to the DB with `push` or `push-force`
    during development.

    **Scripts:** `push`, `push-force`
  </Card>

  <Card title="api-spec" icon="file-code">
    **`lib/api-spec`** · `@workspace/api-spec`

    Owns `openapi.yaml` (OpenAPI 3.1) and the Orval codegen configuration.
    Running `codegen` writes generated output into two sibling packages:
    `api-zod` and `api-client-react`. This is the single source of truth for
    the API contract.

    **Scripts:** `codegen`
  </Card>

  <Card title="api-zod" icon="shield-check">
    **`lib/api-zod`** · `@workspace/api-zod`

    Zod schemas generated from the OpenAPI spec by Orval (e.g.
    `HealthCheckResponse`). Consumed by `api-server` to validate request and
    response shapes at runtime. Do not edit the files under `src/generated/`
    by hand — regenerate them via `api-spec`.
  </Card>

  <Card title="api-client-react" icon="react">
    **`lib/api-client-react`** · `@workspace/api-client-react`

    React Query hooks and a typed fetch client generated by Orval from the
    OpenAPI spec (e.g. `useHealthCheck`, `healthCheck`). Intended for use in
    frontend packages. Like `api-zod`, its `src/generated/` directory is
    machine-written.
  </Card>

  <Card title="scripts" icon="terminal">
    **`scripts`** · `@workspace/scripts`

    Utility and maintenance scripts. Each file in `src/` has a corresponding
    entry in `package.json` `scripts`. Run any script with:
    `pnpm --filter @workspace/scripts run <script-name>`. Scripts can import
    any workspace package as a regular dependency.
  </Card>
</CardGroup>

## Package dependency graph

```text
openapi.yaml  (api-spec)
     │
     │  Orval codegen
     ▼
 ┌─────────────────────┐
 │  api-zod            │  ← Zod schemas
 │  api-client-react   │  ← React Query hooks + fetch client
 └─────────────────────┘
         │                       ┌──────┐
         └──────────────────────▶│  db  │
                  ▼              └──────┘
            api-server ◀─────────────┘
```

## Data flow: OpenAPI spec to runtime

1. **Define** — the API contract lives in `lib/api-spec/openapi.yaml` as a
   single OpenAPI 3.1 document.
2. **Generate** — running `pnpm --filter @workspace/api-spec run codegen`
   invokes Orval, which reads the spec and writes:
   - Zod schemas into `lib/api-zod/src/generated/`
   - React Query hooks + fetch functions into `lib/api-client-react/src/generated/`
3. **Validate** — `api-server` imports schemas from `@workspace/api-zod` to
   validate incoming requests and outgoing responses against the same contract
   defined in the spec.
4. **Consume** — frontend packages import hooks from `@workspace/api-client-react`
   to call the API with full type safety, no manual type definitions required.

<Info>
  Always run `codegen` after editing `openapi.yaml` to keep the generated
  packages in sync. The generated files are committed to the repository so that
  consumers do not need to run codegen on every install.
</Info>

## TypeScript project references

Every package extends `tsconfig.base.json`, which sets `composite: true`.
Cross-package type resolution depends on the full build graph being up to date.

- **Always typecheck from the root**: `pnpm run typecheck`
- Running `tsc` inside a single package may fail if its workspace dependencies
  have not been built yet.
- `tsc` emits `.d.ts` declaration files only (`emitDeclarationOnly`); actual
  bundling is handled by esbuild (`artifacts/`) or tsx (scripts).
